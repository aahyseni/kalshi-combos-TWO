## 2. Client construction & auth [wheel]
```python
from kalshi_python_sync import Configuration, KalshiClient   # or kalshi_python_async

config = Configuration(host="https://external-api.demo.kalshi.co/trade-api/v2")  # demo
config.api_key_id = "your-api-key-id"          # set as ATTRIBUTES —
config.private_key_pem = private_key_pem_str   # NOT constructor kwargs
client = KalshiClient(config)
```
- `Configuration.__init__` does **not** accept `api_key_id`/`private_key_pem` params — they must be set as attributes afterward; `KalshiClient.__init__` checks `hasattr(configuration, 'api_key_id') and hasattr(configuration, 'private_key_pem')` and builds a `KalshiAuth` from them. If unset, requests go out unsigned (no error until 401).
- `Configuration` also accepts `retries: Optional[int]` (async: feeds `aiohttp_retry.RetryClient` with `ExponentialRetry`; see open questions re: retrying POSTs), `ssl_ca_cert`, `cert_file`, `key_file`, `debug`.

### Request signing (`kalshi_python_sync/auth.py`, class `KalshiAuth` [wheel])
- Headers on every authed request:
  - `KALSHI-ACCESS-KEY`: API key ID
  - `KALSHI-ACCESS-TIMESTAMP`: `str(int(time.time() * 1000))` — **milliseconds**, local clock
  - `KALSHI-ACCESS-SIGNATURE`: base64 of RSA-PSS signature
- Message signed: `f"{timestamp}{METHOD.upper()}{path}"` where path **excludes query string** (e.g. `1703123456789GET/trade-api/v2/portfolio/balance`).
- Algorithm: RSA-PSS, `padding.PSS(mgf=padding.MGF1(hashes.SHA256()), salt_length=padding.PSS.DIGEST_LENGTH)`, digest SHA256, then `base64.b64encode`.
- Key must be an RSA key (`load_pem_private_key`, `isinstance(loaded_key, RSAPrivateKey)` enforced).
- `KalshiClient.update_params_for_auth` builds the signed path as `full_path = f"/trade-api/v2{resource_path}"` — the `/trade-api/v2` prefix is **hardcoded** regardless of configured host [wheel]. Fine for all four official hosts; breaks if you ever proxy under a different path prefix.

### KalshiClient dispatch [wheel]
`KalshiClient(ApiClient)` instantiates all 17 API classes (`_account_api, _api_keys_api, _communications_api, _events_api, _exchange_api, _fcm_api, _historical_api, _incentive_programs_api, _live_data_api, _market_api, _milestone_api, _multivariate_api, _order_groups_api, _orders_api, _portfolio_api, _search_api, _structured_targets_api`) and forwards unknown attribute access via `__getattr__` scanning that list in order (linear `hasattr` scan per call — cache bound methods in the hot path, or hold a direct `CommunicationsApi(client)` reference).

Async client: `async def __aenter__/__aexit__/close` on `ApiClient` [wheel] — use `async with` or `await client.close()`. Session is `aiohttp.ClientSession(connector=aiohttp.TCPConnector(limit=maxsize, ssl=...))`.

Every operation has 3 variants: `foo()` (convenience, returns `.data`), `foo_with_http_info()` (typed request model, `@validate_call`, returns `ApiResponse`), `foo_without_preload_content()`.

**TRAP — convenience kwargs wrappers**: `create_quote(**kwargs)` does `CreateQuoteRequest(**kwargs) if kwargs else None`, i.e. ALL kwargs (including `_request_timeout`, `_headers`, ...) are fed into the pydantic model constructor, while options are separately read via `kwargs.get('_request_auth')` etc. Passing underscore options to the convenience method risks pydantic validation errors. For the bot, prefer `create_quote_with_http_info(CreateQuoteRequest(rfq_id=..., yes_bid=..., no_bid=..., rest_remainder=...))`.

### Fixed-point conventions (verbatim from model docs [wheel])
- `contracts_fp`: "Fixed-point contract count string (2 decimals, e.g., \"10.00\"...). Requests accept 0–2 decimal places...; responses always emit 2 decimals. Fractional contract values (e.g., \"2.50\") are supported on markets with fractional trading enabled; the minimum granularity is 0.01 contracts. Integer contract count fields are legacy and will be deprecated; when both integer and fp fields are provided, they must match."
- All dollar amounts: fixed-point decimal **strings**, up to 6 decimal places. Legacy `get_balance()` still returns integer cents (`balance.balance / 100` in README example) — mixed conventions across old vs new endpoints; parse dollars fields with `Decimal`, never float.

## 4. WebSockets — NOT in the SDK; hand-roll (from getting_started/quick_start_websockets.md + websockets/communications.md)
- Prod: `wss://external-api-ws.kalshi.com/trade-api/ws/v2`; Demo: `wss://external-api-ws.demo.kalshi.co/trade-api/ws/v2`. Legacy `wss://api.elections.kalshi.com/trade-api/ws/v2` and `wss://demo-api.kalshi.co/trade-api/ws/v2` still supported.
- Same 3 auth headers on the HTTP upgrade; signature message = `timestamp + "GET" + "/trade-api/ws/v2"`.
- Docs use the Python `websockets` library with `additional_headers`; library's automatic ping/pong satisfies keep-alive ("no manual heartbeat implementation needed").
- Subscribe: `{"id": 1, "cmd": "subscribe", "params": {"channels": ["ticker"], "market_tickers": ["KXFUT24-LSV"]}}`. Errors: `{"id": 123, "type": "error", "msg": {"code": X, "msg": "description"}}`.
- Channels — private: `orderbook_delta`, `fill`, `market_positions`, `communications`, `order_group_updates`; public: `ticker`, `trade`, `market_lifecycle_v2`, `multivariate_market_lifecycle`, `multivariate`.
- `communications` channel: market filter in subscription is **ignored**; optional sharding `shard_factor` (1–100) / `shard_key` (0 ≤ key < shard_factor). Message types: `rfq_created`, `rfq_deleted` (always sent to all subscribers), `quote_created`, `quote_accepted`, `quote_executed` (quote events **only** if you created the quote OR the RFQ). Fields are dollars-strings/fp-strings: e.g. rfq_created msg: `id, creator_id, market_ticker, event_ticker, contracts_fp, target_cost_dollars, created_ts`; quote_accepted msg adds `accepted_side` ("yes"/"no"), `contracts_accepted_fp, yes_contracts_offered_fp, no_contracts_offered_fp, rfq_target_cost_dollars`; `quote_executed` msg: `quote_id, rfq_id, quote_creator_id, rfq_creator_id, order_id, client_order_id, market_ticker, executed_ts` — use `client_order_id` to correlate with `fill` channel messages.

## 6. VERDICT for the maker bot
**Use `kalshi_python_async` for the REST side; hand-roll the WebSocket client.** Rationale:
- The async SDK fully covers the Communications API (create/delete quote, confirm, get RFQs, `GET /communications/id`) with correct built-in RSA-PSS signing, typed pydantic models, and aiohttp — adequate for quote-create/confirm REST calls even in a 3-second HVM confirmation window (one RSA sign + one HTTPS round trip either way).
- The SDK has **zero WebSocket support**, and the `communications` WS channel is the latency-critical RFQ-discovery path — that must be hand-rolled with `websockets` + the same `KalshiAuth` signing (you can import and reuse `kalshi_python_async.auth.KalshiAuth.create_auth_headers("GET", "/trade-api/ws/v2")`).
- Acceptable to hand-roll REST too (it is ~60 lines with `aiohttp` + `cryptography`) if the `>=3.13` Python pin, weekly-regenerated releases, or pydantic validation overhead are dealbreakers; but there is no functional gap forcing it.
- Hard rules if using the SDK: pin `==3.22.0`, import from `kalshi_python_sync`/`kalshi_python_async` (NOT `kalshi_python`), set `api_key_id`/`private_key_pem` as Configuration attributes, prefer `_with_http_info` typed variants over `**kwargs` convenience methods, do not enable `Configuration(retries=...)` until verified it can't duplicate POST /communications/quotes.
